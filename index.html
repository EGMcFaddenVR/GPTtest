1  Introduction to Data Export API

Welcome to the Qu Data Export API. This document details the interfaces and standard use cases for integration 
partners looking to leverage general data from Qu.

The Data Export API is a read-only API that enables partners to retrieve raw data for analytics and post-processing 
for the following core data sets:

Check data to retrieve check-level data including:

Item sales
Service charges
Taxes
Payments
Voids and refunds

Configuration data to retrieve definitions and listings of major data sets including:

Menu item, portion, groupings and category details
Ancillary data for taxes, payments, service charges and location details
Employee data for job , payroll profile and labor hours

Till Reconciliation, Paid in/out, Over/Under data and other financial balancing data
Employee/Labor data for retrieving employee information and time clock data

The high level functions listed above encompass most of the currently available data, but more data points are 
added frequently (or removed much less frequently) in any given release, so it is important to continually keep up 
to date with release notes and check for changes to the data output.

1.1.2 

Request and response structure

The Qu API is organized around REST with predictable, resource-oriented URLs and HTTP response codes to 
indicate API errors. Every request to Qu is comprised of HTTP Headers, endpoint URL and (for some requests) a 
JSON Body. All JSON structures follow the rules specified in RFC4627; and are encoded as UTF8 as specified in 
RFC3629 using one to four bytes per Unicode code point.

All responses return in JSON format and uses standard HTTP response codes for ease of development.

1.3 

Endpoint and parameters

There is only 1 endpoint for the Data Export API, and all data requests are made using parameters to identify the 
sub-type and filtering properties to receive.

 

companyId

 is the customer's unique account ID in Qu, which will always be the same for any given 

location.

locationId

 is the unique location Id (store) for which the data will be retrieved. These location IDs can 

be provided by the brand or Qu account representative or can be 

retrieved using the API

 itself.

Item: “Mac & Cheese”

'config/item_group' and 'checks'  will show only the new ids.  Also if the items 
do not have portions they will be converted as well creating new item_ids.

As such, high level use cases for each major category have been created along with the spec itself. It is up to the 
integrator to responsibly manipulate the data themselves as the exported data provided from this API is raw and 
does not include summaries or aggregations.

start_date

(Basic)

Specifies the starting point for the export. This will include all checks for this date and 
onward until end_date (if applicable).

This is mandatory if not using the delta parameters.

Format: MMDDYYYY 

end_date

(Basic)

Specifies the end date for the export. If applied this will be the end of the date range and 
include all checks up to and including this date.

If end_date is not specified then the export will return all indexed checks up until the 
moment that the request to the API was made.

Format: MMDDYYYY

delta_from_date

(Advanced)

These parameters should only be used when doing incremental export updates 
throughout the day such as for integrators updating data live. If using delta parameters, 
do not use the startDate and endDate parameters.

Format: MMDDYYYY / HHMM

delta_from_time

(Advanced)

include_abandoned

(Advanced)

IncludeAbandoned parameter is only used when requesting checks (data_type=checks). 
When set to "true", the response will include all the checks with state “Abandoned” for 
the requested location and date ranges.

All shifts, tills and checks that were left open by the employees will be processed by an 
automatic task that runs everyday after the business hours of each location. This 
process can be enabled or disabled depending on the customer preferences.

Checks that are processed by this task are set to state "Abandoned".

Format: true, false

Using the delta fields will return data that can be “volatile” in that the check 
data could be altered after the initial polling and should always be reconciled 
with a standard lookback request to ensure the data is complete.

When requesting from a single date or a range of dates, the most recent “iteration” of the check data will always be 
returned. This is an especially important concept to understand when dealing with both basic and advanced 
strategies and is best illustrated with an example.

Scenario: a check is created on September 1st with 3 items. The following day (9/2), the check is re-opened and 2 of 
the items are refunded. The day after that (9/3), the check is re-opened again and the last item is refunded, resulting 
in a $0 check.

Using Strategy 1:

If the application fetches all checks for starting and ending date of 9/1: the check will have 3 items on 
it and no refunds logged
If the application fetches all checks for starting and ending date of 9/2: the check will appear with a 
refund of two items (and not including the 3rd item that was still settled)
If the application fetches all checks for the starting and ending date of 9/3: the check will appear with 
the last item refunded

Using Strategy 2:

If the application fetches all checks for the range of 9/1 - 9/2: the check will appear with a refund of 
two items (and not including the 3rd item that was still settled)
If the application fetches all checks for the range of 9/1 - 9/3: the check will appear with all items 
refunded

Why is it like this?

When a check is created in the system, it is immediately saved to the cloud as is. When changes occur, a new version 
of the check is published and the previous version is logged as a child data set within the check. This allows for 
consistency in daily reporting so it is possible to look up a previous day and see what the exact conditions were at 
the time of closing on that day, without conflating it with changes that occurred later on.

Get incremental check data throughout the day

The Qu Data Export API does not currently support data-streaming, so all intra-day updates have to be polled from 
the integrator-side. While there are no hard limits on the frequency, It is generally recommended to set this cadence

Track abandoned checks

Abandoned checks are defined within Qu as a check that has been started and has content on it but has not been 
paid for by the time the end of day report is run. When the EOD process runs, it will force any unpaid checks into the 
abandoned state to remove them from the suspended check list so that no checks remain unaccounted for in the 
check data.

How are abandoned checks created?

These checks are created when a check is suspended by the cashier, either as a “shortcut” to voiding all the items 
on an order cancelled before being fully paid or as a way to put a check aside while a guest goes to do something 
else such as get their wallet from the car. Occasionally it can also indicate training issues or worse, theft, since the 
check can still be sent to the kitchen for processing.

By default, abandoned checks are filtered out in the Data API export or in Qu’s EI reporting views as they do not 
count as paid and closed checks. To keep them in results, include the parameter “include_abandoned=true” in the 
API request.

The use of this parameter to generate these unique IDs is consistent and will not change over time for any 
specific item x portion mapping unless one of those IDs is altered.
However the ID is not currently available for review within the Qu platform, the structure only exists in the 
Data Export API translation layer.

The Qu system lists deposits based on the date of creation in the system, not the date of the actual deposit to the 
bank. This is because deposits may be made to the bank in non-consecutive days and requires user entry to 
confirm. For example, some businesses may add the deposits the next day everyday except Sunday, and some may 
log them right before the end of a pay period. This is important to note as the parameters for start and end date 
will return the deposits created in the time period, not necessarily the deposits made in that time period.
